# code-review

任意のスコープに対する建設的なコードレビューを行う。レビュアーの目線でコードを読み、**実際に意味のある問題**を優先度順に報告する。スタイルや好みの押し付けはしない。

質の担保は **観点別 finder の並列 fan-out → 指摘ごとの検証・confidence 採点 → 閾値フィルタ** で行う（プラグインの `/review` と同じレシピを、PR 依存をなくして任意スコープに一般化したもの）。単発の順次チェックはリスクの高い変更を見落としやすいため、非自明な差分では必ず fan-out する。

## レビュー対象の決定

ユーザーの指示と引数から、レビュー対象を確定する。曖昧な場合は推論し、対象を 1 行で明示してからレビューに入る。

| ユーザーの言い回し / 引数 | 対象 |
|---|---|
| 「現在のブランチ」「この変更」 引数なし | `git diff <base>..HEAD` + 未コミット変更 |
| `--staged` / 「ステージしたやつ」 | `git diff --cached` |
| ファイルパス | そのファイル全体 |
| `HEAD~3` / `HEAD~5..HEAD` | そのコミットレンジ |
| `origin/main..HEAD` 等 | 指定範囲の差分 |
| `#123` / `PR 123` / PR の URL | `gh pr diff 123` で差分を取得し、以降はローカル差分と同じ扱いでレビュー |

base ブランチは以下の順で推定: 引数 → `git symbolic-ref refs/remotes/origin/HEAD` → `main` → `master` → `develop`。

PR をレビューする場合でも、**GitHub 上にコメントは投稿しない**（それは `/review` の役割）。明示的に「PR にコメントして」と言われない限り、結果は会話に返す。PR の場合のみ finder F6（過去 PR コメント）が追加で有効になる。

## 事前準備（必須）

1. **プロジェクト規約の把握**: `CLAUDE.md`、`AGENTS.md`、`.github/PULL_REQUEST_TEMPLATE.md`、`README.md` を該当範囲で確認する。プロジェクト固有のルールを優先する。
2. **差分の取得**: `git diff <range>` で全体差分を取り、`git diff --stat` でファイル粒度の変更規模を把握する。
3. **コンテキストの取得**: 差分だけでは判断できない場合、変更されたファイルの周辺・呼び出し元・型定義を読む。**差分外の文脈を読まずにコメントしない。**

## レビュー観点（優先度順）

### 1. 正しさ（Correctness）
- ロジックバグ、オフバイワン、null/undefined ハンドリング漏れ
- 競合状態、並行処理の問題
- 型の不整合、暗黙の型変換による事故
- エラーハンドリング漏れ、握り潰し
- エッジケース（空配列、最大値、Unicode、タイムゾーン、負数 など）
- **終端性・状態遷移（最重要リスク）**: ループ条件 / ポーリング述語 / リトライ・バックオフ / 再帰の停止条件 / `while`・`useEffect`・購読の解除条件を変更した場合は、必ず「**どの状態でこれが停止しないか**」を明示的に答える。enum / status を条件に使う変更は、**全ての値を列挙し、変更後のロジックと消費側（表示・分岐）のロジックの両方に通して**、2 つの契約が一致するかを確認する（例: `status` の各値 ×「ポーリングする?／表示する?」の対応表を作ると、片方だけ真になる矛盾が浮く）。

### 2. セキュリティ
- 入力検証漏れ（SQL/コマンド/パスインジェクション、XSS、SSRF）
- 認証・認可の抜け
- シークレット・トークンの平文ログ出力やコミット
- 安全でない暗号、弱い乱数

### 3. 仕様・契約
- 既存の API/関数のシグネチャ変更による破壊的影響
- 後方互換性、マイグレーション漏れ
- 設定ファイル・スキーマ・テンプレート間の整合性
- ドキュメントとの乖離
- **兄弟実装との契約整合**: 同種のパターン（同じ composable ファミリ、同じ Hook、同じ base クラスの他実装）を grep し、変更後のコードがその確立された契約に一致するかを確認する。スタイルの一貫性ではなく「正しい振る舞いの契約」を兄弟から復元する。

### 4. テスト
- 重要な分岐がテストされているか
- テストが本物の振る舞いを検証しているか（ただのスナップショット/モック確認になっていないか）
- フレイキー要因（時間、ランダム、外部依存）
- ハッピーパスのみで終わっていないか

### 5. 可読性・保守性
- 命名、関数のスコープ、責務の分離
- 過剰な抽象化／早すぎる一般化
- 重複コード、Magic Number
- コメントが「なぜ」を語っているか（「何を」を冗長に書いていないか）

### 6. パフォーマンス
- 明らかな N+1、不要なループ・コピー
- ホットパスでのアロケーション
- 不必要な同期 I/O、ブロッキング
- ※ 計測根拠のないマイクロ最適化は提案しない

## 検出プロセス（fan-out → verify → synthesize）

これが本スキルの中核。上記「レビュー観点」を単一パスで通すのではなく、観点をばらして並列に走らせ、各指摘を独立に検証してから統合する。

### Phase 1: fan-out（観点別 finder を並列 spawn）

差分の規模に応じて **4〜6 個の finder サブエージェント**を Agent / Task ツールで**同時に** spawn する（1 メッセージ内で複数 spawn）。各 finder にはレビュー対象の range と、下記の担当レンズだけを渡し、**指摘の一覧**を構造化して返させる。各指摘は `{ file:line, 要約, 根拠(参照した行・履歴・兄弟実装), 想定失敗シナリオ }` を必須項目とする。

- **F1 規約準拠**: `CLAUDE.md` / `AGENTS.md` / 近傍のコード内コメントに反していないか。規約が**その箇所を明示的に指している**場合のみ指摘する（一般論の押し付けはしない）。
- **F2 バグ走査**: 差分そのものを中心に、素朴なバグを走査。大物優先。nit・linter/型検査で機械的に捕まるもの・明白な誤検知は出さない。
- **F3 履歴文脈**: `git blame` / `git log -p` で変更行の来歴を読み、過去の意図・過去の修正を壊していないか（一度直したバグの再導入など）を確認。
- **F4 状態・制御フロー**: ループ終端 / ポーリング・リトライ条件 / async・並行 / enum・status の網羅性。「どの状態で停止しないか」「全状態を変更後ロジックと消費側の両方に通したか」を必ず回答に含める。**前回この観点の欠落で P0 を見落としたため、独立 finder として常設する。**
- **F5 契約・整合**: 呼び出し元 / 型 / スキーマ / 設定 / 兄弟実装との齟齬。同種パターンの既存実装を grep して契約を復元し、変更がそれに一致するかを検証。cross-file の破壊（rename の取り残し、stale 参照）も担当。
- **F6 過去 PR コメント（PR レビュー時のみ）**: 当該ファイルを触った過去 PR のレビューコメントを `gh` で参照し、同種の指摘が再発していないか確認。

finder が使えない環境（Agent/Task ツール不可）では、同じレンズを自分で順に通す。ただしその場合も各レンズを**独立した観点として明示的に**走らせ、混ぜない。

### Phase 2: verify（指摘ごとの反証・confidence 採点）

Phase 1 で集まった全指摘（重複は統合）に対し、**1 件ずつ**検証する。可能なら検証も並列サブエージェントに投げる。各指摘について:

1. **反証を試みる**: 「これは実際には問題ない」と示す反例を 1 つ構築しにいく。反例が作れたら破棄。
2. **失敗シナリオを具体化する**: `<具体的な入力・状態> → <誤った出力・挙動>` を書けるか。書けない指摘は Critical / Important に上げない。
3. **confidence を 0〜100 で採点する**（下記ルーブリック）。

confidence ルーブリック:
- **0**: 全く自信なし。軽い精査で崩れる誤検知、または差分外（既存）の問題。
- **25**: ある程度。実問題かもしれないが誤検知の可能性もあり、実在を検証できていない。規約由来なら、その規約が当該箇所を明示していない。
- **50**: 中程度。実在は確認したが nit か稀なケースで、PR 全体では重要度が低い。
- **75**: 高い。二重チェックし、実際に踏まれる可能性が高い実問題。既存アプローチでは不十分。機能に直接影響、または規約の明記事項。
- **100**: 確実。踏まれる頻度も高く、証拠が直接裏付ける。

**フィルタ**: Critical / Important に載せるのは **confidence 80 以上、かつ具体的失敗シナリオを書けたもの**のみ。差分外（既存）の問題、誤検知、linter/型検査級、下記「何をレビューしないか」該当は捨てる。Suggestion / Nits はこの 80 ゲートを通さないが、最小限・キャップ内に留める（埋め草にしない）。

### Phase 3: synthesize（統合して出力）

生き残った指摘を優先度順に並べ、下記フォーマットで報告する。fan-out した事実は書かなくてよいが、確信度が中程度の指摘には「〜であれば問題」と条件を明示する。

## レビューの心得

- **シカトしない、忖度もしない**: 重要な問題は遠慮なく指摘する。逆に「念のため」「気になっただけ」レベルのコメントは出さない。
- **「問題なし」を安売りしない**: リスクの高い変更（条件・状態遷移・async・終端性）に「整合している」「破綻はない」と書くときは、**状態を実際にトレースした根拠**（対応表・全状態の追跡）を持っていること。反例を探して見つからなかったときだけ「問題なし」と断定する。
- **失敗シナリオで語る**: Critical / Important は必ず「どの入力・状態でどう壊れるか」を伴う。書けないなら格を下げるか出さない。
- **Nit を埋め草にしない**: スタイル・命名の好みは Nit としてまとめ、最大数件まで。Linter で機械的に解決すべき指摘はしない。
- **行番号で指摘する**: `path/to/file.ts:42` の形式。レビュアーが即座にジャンプできるようにする。
- **解決策を提示する**: 問題だけ並べない。「こう直す」を 1〜数行で示す。曖昧な場合は選択肢を提示。
- **既存コードとの一貫性・契約**: 周辺の慣習・兄弟実装の契約に従っているかを必ず確認。「正解」でも周辺と食い違うなら触れる。
- **推測は明示する**: 「〜の可能性がある」「〜であれば問題」と書く。断定するなら根拠（行番号、参照先）を添える。
- **既に正しいものは褒めない、流す**: 「Good」「LGTM」等のコメントで埋めない。問題がなければそのセクションを書かない。

## 出力フォーマット

レビュー結果は次の構造で報告する。該当がないセクションは省略する。

```markdown
## レビュー対象
<range / file / PR を 1 行で>

## 重大な問題 (Critical)
バグ、データ破壊、セキュリティ事故、本番障害を引き起こすもの。マージ前に必須対応。

- `path/file.ts:L42` — 〈問題の要約〉
  〈詳細：何が、なぜ問題か〉
  **失敗シナリオ**: 〈具体的な入力・状態 → 誤った挙動〉
  **修正案**: 〈具体的な修正〉

## 重要な指摘 (Important)
誤った振る舞い、契約違反、テスト不足など、修正すべきもの。

## 改善提案 (Suggestion)
より良い設計・可読性・保守性のための提案。任意対応。

## Nits
スタイル・命名の細かな指摘。最大 5 件まで。

## 良かった点
**特に非自明な良い判断があった場合のみ** 1〜2 項目記載。義務的に書かない。
```

## 何をレビューしないか / 誤検知の例

以下は指摘しない。finder / verifier もこれらを誤検知として扱う:

- フォーマッタ・Linter・型検査・コンパイラで機械的に捕まるもの（インデント、import 順、行末スペース、型エラー、壊れたテスト）。CI で別途走る前提でよい。
- 差分外（既存）の問題。変更行がその箇所を悪化させていない限り触れない。
- 「個人的に好み」レベルの命名変更、senior が指摘しない衒学的な nit。
- 触れていないコードのリファクタ提案（スコープ外）。
- 既知の TODO や負債（差分でその箇所が悪化していない限り）。
- バグに見えて実はバグでないもの、意図的と思われる／広い変更に直接関連する挙動変更。
- 規約に反するが、コード側で明示的に抑止されているもの（lint ignore コメント等）。
- テスト網羅・一般的なセキュリティ・ドキュメント不足などの一般論（規約で明示的に要求されている場合を除く）。

## 実行フロー

### Step 1: スコープと差分の取得
```bash
git branch --show-current
git status
git diff --stat <range>
git diff <range>
```
PR の場合は `gh pr diff <PR>` / `gh pr view <PR>` で差分と説明を取得する。

### Step 2: プロジェクト規約の確認
```bash
ls CLAUDE.md AGENTS.md .github/PULL_REQUEST_TEMPLATE.md 2>/dev/null
```
存在するものと、変更ファイルのディレクトリ直下の `CLAUDE.md` を読む。

### Step 3: 必要な周辺コードの読み込み
差分から呼び出される / 呼び出す側、型定義、設定ファイル、兄弟実装など、判断に必要な範囲を Read する。

### Step 4: fan-out（Phase 1）
差分規模に応じて 4〜6 個の finder を並列 spawn（些細な差分は Step 4.5 の fast path へ）。

### Step 5: verify（Phase 2）
集まった指摘を統合し、1 件ずつ反証・失敗シナリオ化・confidence 採点。80 未満と差分外を捨てる。

### Step 6: 出力（Phase 3）
上記フォーマットで報告。生き残った Critical / Important が無ければ「重大な問題はなし。〜」と 1〜2 行で結論を述べ、それでも気づいた小さな提案があれば Suggestion / Nits に記載。

## 特記事項

- **言語**: ユーザーが日本語で依頼している場合、レビューも日本語で書く。コード片はそのまま英語で OK。
- **規模に応じたスケール（fast path）**: 差分が些細（おおむね 30 行未満・単一の明白な変更、設定 1 行など）なら fan-out せず単発精読でよい。ただし状態遷移・条件・async・終端性に触れる変更は規模に関わらず F4 の観点だけは必ず通す。
- **大規模差分**: 500 行超 / 20 ファイル超では、まずファイル単位で重要度を見積もり、リスクの高いファイルに finder を厚く割り当てる。すべて等しく深掘りしない。
- **生成コード・ベンダーファイル**: `dist/`、`vendor/`、`*.lock`、自動生成ファイルはスキップする（差分があってもレビュー対象外と明記）。
- **不明点はユーザーに聞かない**: 1 ファイル読めば分かることは読む。本当に意図が不明な箇所だけ、「〜の意図を確認したい」と末尾でまとめて聞く。
