# pr-comment-reply

自分が出した PR に付いたレビューコメントへの返信を作成する。返信は必ずドラフトとして提示し、ユーザーの承認を得てから `gh` で投稿する。

## 大原則

- **すぐ送らない**: 返信文を作ったら、そのまま投稿せず、まずユーザーに見せて確認を取る。承認が出てから投稿する。
- **簡潔に**: 詳細を全部書かない。本質的な情報だけを一文〜数文で。冗長な説明・言い訳・水増しはしない。
- **メンションの使い分け**（重要、間違えやすい）:
  - **対応したとき（修正コミットで解決）→ メンションを付けない**。相手にアクションを求めていないため。
  - **質問を返す / 対応しないとき → メンションを付ける**（`@reviewer`）。相手の反応が必要なため。

## 返信のパターン

コメントへの返信は次の 3 パターンに分かれる。まずどれに当たるかを判断する。

### パターン A: 修正コミットを追加して対応した

コメントの指摘を受けて修正し、そのコミットで解決した場合。**メンションなし**。

返信先によって文面が変わる:

- **インラインコメントにスレッドで返信するとき**（コード上の特定行への指摘）:
  ```
  [コミットハッシュ] で対応しました
  ```
  例: `a1b2c3d で対応しました`

- **通常コメント（会話タブ / PR 全体へのコメント）に返信するとき**:
  ```
  以下を対応しました
  [コミットハッシュ] 対応内容を一文で
  ```
  複数コミットなら行を増やす。例:
  ```
  以下を対応しました
  a1b2c3d バリデーションエラー時のメッセージを追加
  e4f5g6h 早期 return で nil チェックを整理
  ```

**「対応内容を一文で」の書き方**: コミットメッセージと `git show <hash>` の diff を読み、何をしたかを一文で要約する。コミットメッセージが既に的確ならそれを使ってよいが、指摘への回答として不自然なら diff を見て言い換える。命令形の過去表現（「〜を追加」「〜を修正」）で、レビュアーが指摘との対応を一目で追えるようにする。

### パターン B: 質問を返す

指摘の意図が不明、または対応方針を相談したいとき。**メンションあり**。

```
@reviewer <簡潔な質問>
```

例: `@alice ここは意図的に同期処理にしています。非同期にすべきという指摘でしょうか？`

### パターン C: 対応しない

指摘は理解したが、あえて対応しない場合。**メンションあり**。理由を簡潔に一文で。

```
@reviewer <対応しない理由を簡潔に>
```

例: `@bob この挙動は後方互換のため意図的に残しています。別 Issue で追う想定です。`

## 実行フロー

### Step 1: PR とコメントの特定

対象 PR 番号が引数 / 会話で分かればそれを使う。不明なら現在のブランチから特定する:

```bash
gh pr view --json number,url,headRefName
```

コメントを取得する。**インラインコメント（レビューコメント）と通常コメント（issue コメント）は API が別**なので区別する:

```bash
# インラインコメント（コード行への指摘）: id, path, line, body, user を確認
gh api repos/{owner}/{repo}/pulls/{number}/comments

# 通常コメント（会話タブ）
gh api repos/{owner}/{repo}/issues/{number}/comments
```

`owner/repo` は `gh repo view --json owner,name` で取得できる。どのコメントに返信するか曖昧なら、ユーザーに一覧を見せて対象を確認する。

### Step 2: パターン判定と対応コミットの特定

各コメントについて A / B / C のどれかを判断する。パターン A の場合、どのコミットが対応に当たるかを特定する:

```bash
git log origin/{base}..HEAD --oneline   # 最近のコミット一覧
git show <hash>                          # 内容確認（一文要約の材料）
```

どのコミットが指摘に対応するか自明でなければ、ユーザーに確認する。

### Step 3: 返信ドラフトの作成と確認

各コメントに対する返信文を作り、**投稿せずに** ユーザーへ提示する。以下を明示する:

- どのコメント（誰の、どの指摘）への返信か
- パターン（A/B/C）とメンションの有無
- 返信本文

ユーザーの承認・修正を待つ。

### Step 4: 投稿

承認された返信のみ投稿する。

- **インラインコメントへのスレッド返信**（`in_reply_to` にコメント ID を指定）:
  ```bash
  gh api repos/{owner}/{repo}/pulls/{number}/comments \
    -f body='a1b2c3d で対応しました' \
    -F in_reply_to=<comment_id>
  ```

- **通常コメント（会話タブ）への返信**:
  ```bash
  gh pr comment <number> --body "$(cat <<'EOF'
  以下を対応しました
  a1b2c3d バリデーションエラー時のメッセージを追加
  EOF
  )"
  ```

投稿後、返信した URL / 結果をユーザーに報告する。

## チェックリスト（投稿前に自問する）

- [ ] 対応した返信（パターン A）にメンションを付けていないか？
- [ ] 質問 / 非対応（パターン B・C）にメンションを付けているか？
- [ ] インライン指摘へは `in_reply_to` でスレッド返信しているか？（新規トップレベルコメントになっていないか）
- [ ] 一文要約が指摘への回答として的確で、簡潔か？
- [ ] ユーザーの承認を得たか？（未承認のまま投稿していないか）

## 特記事項

- **勝手にコミットしない**: このスキルはコメント返信が責務。修正コミット自体はユーザーが作る（または別途依頼される）前提。存在するコミットを参照して報告するだけ。
- **コミットハッシュ**: 短縮ハッシュ（7 桁程度）で十分。GitHub 上でリンクになる。
- **複数コメントの一括処理**: 「全部のコメントに返信して」と言われた場合も、パターン判定 → ドラフト一覧提示 → 一括承認 → 投稿、の順を守る。承認前に投稿しない。
- **言語**: 特に指定がなければ、元コメントやリポジトリの言語に合わせる。上記の日本語テンプレートは一例。
