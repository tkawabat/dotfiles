対象: $ARGUMENTS

上記の PR レビューコメントについて、**それが正しいかどうかをフラットに判断**し、対応要否と（必要なら）方針を返す。

## 原則: レビュアーが正しい前提で動かない

- コメントの主張を **現在のコードで検証** する。「指摘されたから直す」ではなく、主張が事実として成り立つかを確かめる。
- 判定は 3 択でフラットに: **正しい / 誤り（事実誤認・前提が古い）/ 部分的に妥当（トレードオフあり）**。
- レビュアーに忖度して過剰に同意しない。逆に、指摘を軽視もしない。コードと根拠で決める。

## 実行フロー

### Step 1: 対象の特定

引数の URL / PR 番号をパースする。GitHub の URL アンカーで種別が分かる:

| アンカー | 種別 | 取得 API |
|---|---|---|
| `#pullrequestreview-<id>` | レビュー（本文＋ぶら下がりコメント群） | `pulls/{pr}/reviews/{id}` と `.../reviews/{id}/comments` |
| `#discussion_r<id>` | インラインコメント単体 | `pulls/comments/{id}` |
| `#issuecomment-<id>` | PR 全体コメント単体 | `issues/comments/{id}` |
| アンカーなし | PR 全体 | 下記でレビュー・コメントを一覧 |

```bash
# owner/repo/pr を URL から取り出して使う。例: stockmarkteam/anews-ui #5122
gh api repos/{owner}/{repo}/pulls/{pr}/reviews/{review_id}
gh api repos/{owner}/{repo}/pulls/{pr}/reviews/{review_id}/comments
# アンカーなしで全体を見る場合
gh pr view {pr} --repo {owner}/{repo} --json title,body
gh api repos/{owner}/{repo}/pulls/{pr}/reviews
gh api repos/{owner}/{repo}/pulls/{pr}/comments
```

取得できない場合（権限・認証）は WebFetch で URL を読むフォールバックを試す。

### Step 2: 重要コメントの選別

レビューに複数コメントがぶら下がる場合、**実質的な指摘に絞る**。typo・好み・nit は要否のみ簡潔に触れ、深掘りしない。単体コメントの URL が渡された場合はそれ 1 件に集中する。

### Step 3: 現状コードの把握

各コメントについて、指している **ファイル・行の現在の内容** と、判断に必要な周辺（呼び出し元・型定義・関連実装）を読む。

- コメントが引用しているコードが **既に変わっている**（後続コミットで対応済み・行がずれている）ことがある。必ず現在の状態を確認する。
- 差分だけで判断できないときは差分外も読む。**文脈を読まずに結論を出さない。**

対象リポジトリがローカルにあればそれを読む。無ければ `gh api .../contents/...` や `gh pr diff` で取得する。

### Step 4: フラットに判定

コメントごとに、主張がコードに対して成り立つかを検証し、次を出す:

- **判定**: 正しい / 誤り / 部分的に妥当
- **根拠**: どのコードのどの挙動からそう言えるか（file:line で示す）

### Step 5: 結論を返す

コメントごとに、以下のどちらかを返す:

- **対応必要** → 方針を提案する。何を・なぜ・どう直すか。選択肢が複数あるならトレードオフ付きで比較し、推奨を先頭に。影響範囲・既存テストへの影響も触れる。
- **対応不要** → 理由を明示する。典型: 既に対応済み / 主張が事実誤認 / トレードオフとして現状が妥当 / スコープ外。

実装まではこの時点では行わない。方針に合意が得られてから着手する。

## 出力フォーマット

```markdown
## 対象
<PR / レビュー / コメントへのリンクと一行要約>

## コメントごとの判定

### 1. <コメントの要旨>
- 判定: 正しい / 誤り / 部分的に妥当
- 根拠: <file:line と挙動>
- 結論: 対応必要（方針: …） / 対応不要（理由: …）

### 2. …
```

複数コメントがあるときの総括（優先度・まとめて対応できるか）が有用なら末尾に添える。
